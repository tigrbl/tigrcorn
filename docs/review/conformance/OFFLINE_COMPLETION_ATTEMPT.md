## What the provisional bundles are

The provisional HTTP/3 bundle is generated by mapping each missing independent HTTP/3 / RFC 9220 scenario to its closest same-stack replay scenario.

The provisional all-surfaces bundle is generated by mapping the stricter all-surfaces-independent profile's RFC 7692 / RFC 9110 / RFC 6960 scenarios to the local conformance corpus.

The provisional flow-control bundle is generated by mapping QUIC / HTTP/3 flow-control review scenarios to the closest same-stack HTTP/3 replay artifacts and linking them back to the local flow-control vectors.

The intermediary/proxy corpus bundles the preserved third-party HTTP/1.1 `curl` artifact together with carrier-specific CONNECT relay vector metadata so the repository no longer relies only on prose to describe proxy-style behavior.

Together they preserve:

- missing scenario directory names
- explicit mapping metadata from the missing independent scenario identifiers to the best available offline source material
- enough bundle shape to document what remains open under both the canonical boundary and the stricter planning overlay

The intermediary/proxy corpus improves repository transparency but it is also not a certification bundle.

They do **not** provide independent-certification evidence.

Each provisional scenario directory contains `provisional_metadata.json`, and each provisional `result.json` is marked with:

- `provisional_non_certifying_substitution: true`
- `release_gate_eligible: false`
- `strict_profile_only: true` when the scenario belongs only to the stricter all-surfaces overlay
- either `source_same_stack_scenario` or `source_local_conformance_vector`

## Why the package is still not honestly certifiable end to end

The preserved third-party HTTP/3 / RFC 9220 artifacts are still missing under the authoritative boundary.

The stricter all-surfaces-independent profile would also still require thirteen additional third-party preserved artifacts for RFC 7692, RFC 9110 CONNECT / trailers / content coding, and RFC 6960.

The offline execution environment used for this update did not contain `aioquic`, so the true third-party client adapters under `tests/fixtures_third_party/` could not be executed to regenerate the canonical independent bundle.

Because of that, the package remains **not yet certifiably fully RFC compliant** even after these updates.

## How to replace the provisional bundles with real evidence

1. Install the certification dependencies:

   `pip install -e ".[certification]"`

2. Regenerate the true third-party HTTP/3 / RFC 9220 bundle:

   `PYTHONPATH=src:. python tools/regenerate_independent_http3_release_bundle.py --promote --enable-matrix`

3. Preserve third-party artifacts for the stricter all-surfaces-independent profile described in `docs/review/conformance/ALL_SURFACES_INDEPENDENT_STATUS.md`.

4. Re-run the release gates and documentation checks:

   `PYTHONPATH=src:. pytest -q tests/test_release_gates.py tests/test_external_independent_peer_release_matrix.py tests/test_provisional_http3_gap_bundle.py tests/test_provisional_all_surfaces_gap_bundle.py tests/test_provisional_flow_control_gap_bundle.py tests/test_intermediary_proxy_corpus.py tests/test_scheduler_runtime.py`

5. Remove or archive the provisional bundles once the real artifacts have been preserved.
